# 3.47 Agent 对话完成通知与超时优化

## 概述

解决 Agent 长时间执行时用户感知不足的问题：对话完成后发送系统级通知和应用内 toast，同时修复超时机制的两个 bug。

## 问题背景

1. Agent 执行工具调用可能需要很长时间（如 deploy 操作 600 秒超时），用户切到后台后不知道对话何时完成
2. 超时被取消时显示"⏹️ 操作已被用户停止"，与用户手动停止的提示一样，造成困惑
3. 超时时间硬编码（Agent 10 分钟、部署 15 分钟），50 轮工具调用很容易超时

## 解决方案

### 1. 双层通知

**系统通知（OS 级）**：通过已有的 `NotificationManager` 发送，覆盖 macOS（osascript）、Windows（Toast）、Linux（notify-send）。在 `runAgentLoop` 的两个正常退出点触发：
- 正常完成（`len(resp.ToolCalls) == 0`）：提示"AI 对话完成"
- 达到最大轮次：提示"AI 对话已结束"

```go
func (a *App) notifyAgentComplete(title, message string) {
    if a.notificationMgr != nil {
        a.notificationMgr.Send(title, message)
    }
}
```

**应用内 Toast**：在前端 `ai-chat-complete` 事件处理中，检查 `visible` prop（AIChat 组件是否在当前视图），不可见时弹出 toast 通知。

```javascript
if (!visible) {
    toast.info(data.success
        ? (t.aiChatCompleteNotify || 'AI 对话已完成')
        : (t.aiChatFailedNotify || 'AI 对话执行失败'));
}
```

### 2. 超时提示区分

`ctx.Err()` 返回两种错误：
- `context.Canceled` → 用户手动停止 → 显示"⏹️ 操作已被用户停止"
- `context.DeadlineExceeded` → 超时 → 显示"⏱️ 操作超时（已执行 N 轮）"

在 `runAgentLoop` 的两个 `ctx.Err()` 检查点（循环前检查 + API 调用报错后检查）都做了区分。

### 3. 动态超时

超时不再硬编码，改为：`基础超时 + maxRounds × 30 秒`

```go
dynamicTimeout := timeout + time.Duration(maxRounds)*30*time.Second
```

| 配置 | 旧超时 | 新超时 |
|------|--------|--------|
| Agent 50 轮 | 10 分钟 | 10 + 25 = 35 分钟 |
| 部署 50 轮 | 15 分钟 | 15 + 25 = 40 分钟 |
| Agent 100 轮 | 10 分钟 | 10 + 50 = 60 分钟 |

## 注意事项

1. 系统通知依赖 `NotificationManager.enabled` 开关（设置页面控制），关闭后不发送 OS 通知，但 toast 不受影响
2. Toast 只在用户不在 AI 对话页面时显示，避免重复提醒
3. `visible` prop 由 App.svelte 通过 `style:display` 控制，AIChat 始终挂载但可能不可见
4. 通知不覆盖 `AIChatStream`（非 Agent 模式），因为这些模式通常几秒内完成，不需要通知
