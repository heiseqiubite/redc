# 后端 i18n 国际化方案设计

## 概述

本文档描述 redc 后端（Go）的国际化（i18n）实现方案，覆盖 CLI 命令行和 GUI 桌面端两种运行模式。该方案与前端已有的 `frontend/src/lib/i18n.js` 互补，共同构成完整的中英文双语支持。

**实现日期**: 2026-03-02  
**涉及版本**: v3.0.6

---

## 1. 架构设计

### 1.1 核心包

位置：`i18n/` 目录，包名 `i18n`，模块路径 `red-cloud/i18n`。

```
i18n/
├── i18n.go    # 核心逻辑：语言切换、翻译查找、环境检测
├── zh.go      # 中文翻译表 (zhMessages map)
└── en.go      # 英文翻译表 (enMessages map)
```

### 1.2 核心 API

```go
package i18n

// Init 初始化语言。传空字符串时自动从环境变量检测。
func Init(lang string)

// SetLang 设置当前语言。仅支持 "zh" 和 "en"。
func SetLang(lang string)

// GetLang 获取当前语言代码。
func GetLang() string

// T 根据 key 返回当前语言的翻译文本。key 不存在时回退到中文，仍不存在则返回 key 本身。
func T(key string) string

// Tf 带格式化参数的翻译，等价于 fmt.Sprintf(T(key), args...)。
func Tf(key string, args ...interface{}) string
```

### 1.3 线程安全

所有 API 通过 `sync.RWMutex` 保护，支持并发调用。GUI 的 Wails goroutine 和 CLI 主线程均可安全使用。

### 1.4 语言检测优先级

`Init("")` 时自动检测，环境变量优先级：

```
REDC_LANG > LC_ALL > LC_MESSAGES > LANG > LANGUAGE
```

- 值以 `zh` 开头 → 中文
- 值以 `en` 开头 → 英文
- 无匹配时默认中文（向后兼容）

## 2. CLI 接入方式

### 2.1 初始化时机

在 `cmd/root.go` 的 `PersistentPreRun` 中调用：

```go
PersistentPreRun: func(cmd *cobra.Command, args []string) {
    i18n.Init("")  // 自动从环境变量检测语言
    // ... 其余初始化逻辑
}
```

### 2.2 使用示例

**cobra 命令描述（init 阶段）：**
```go
var stopCmd = &cobra.Command{
    Use:   "stop [id]",
    Short: i18n.T("stop_short"),
    // ...
}
```

**运行时消息：**
```go
gologger.Info().Msg(i18n.T("init_running"))
gologger.Error().Msgf(i18n.Tf("scene_create_failed", templateName, err.Error()))
```

**Flag 描述：**
```go
rootCmd.Flags().BoolVarP(&showVer, "version", "v", false, i18n.T("flag_version"))
```

### 2.3 用户切换语言

```bash
# 默认跟随系统语言
redc plan ecs

# 强制英文输出
REDC_LANG=en redc plan ecs

# 强制中文输出
REDC_LANG=zh redc plan ecs
```

## 3. GUI 接入方式

### 3.1 语言同步机制

前端调用 `SetLanguage(lang)` 时，后端同时：
1. 持久化到 `gui_settings.json`
2. 调用 `i18n.SetLang(lang)` 同步翻译模块

```go
// app.go
func (a *App) SetLanguage(lang string) error {
    i18n.SetLang(lang)  // ← 同步后端 i18n
    // 持久化保存...
}
```

`GetLanguage()` 也同步设置 i18n：

```go
func (a *App) GetLanguage() string {
    // 加载或检测语言后...
    i18n.SetLang(lang)  // ← 确保后端状态一致
    return lang
}
```

### 3.2 后端消息输出

GUI 后端通过 `runtime.EventsEmit(ctx, "log", message)` 推送日志到前端。这些 message 现在可以使用 i18n：

```go
runtime.EventsEmit(a.ctx, "log", i18n.T("config_load_success"))
```

系统通知（`NotificationManager`）也已接入 i18n：

```go
func (nm *NotificationManager) SendSceneStarted(sceneName string) {
    title := i18n.T("notify_scene_started")
    message := i18n.Tf("notify_scene_started_msg", sceneName)
    nm.Send(title, message)
}
```

## 4. 翻译 Key 命名规范

### 4.1 命名模式

```
{模块}_{描述}
```

**模块前缀：**

| 前缀 | 说明 | 示例 |
|------|------|------|
| `root_` | root 命令 | `root_short`, `root_long` |
| `init_` | init 命令 | `init_short`, `init_running` |
| `plan_` / `run_` | plan/run 命令 | `plan_short`, `run_short` |
| `exec_` | exec 命令 | `exec_short`, `exec_connect_failed` |
| `cp_` | cp 命令 | `cp_short`, `cp_upload_success` |
| `action_` | 通用动作 | `action_failed`, `action_success` |
| `stop_` / `start_` / `kill_` / `rm_` / `ps_` / `status_` / `change_` | 各子命令 | `stop_short` |
| `compose_` | compose 编排 | `compose_up_short`, `compose_down_failed` |
| `logs_` | logs 命令 | `logs_short` |
| `tmpl_` | template 管理 | `tmpl_short`, `tmpl_ls_short` |
| `flag_` | flag 描述 | `flag_version`, `flag_debug` |
| `scene_` | 场景操作消息 | `scene_create_done`, `scene_start_failed` |
| `notify_` | 系统通知 | `notify_scene_started` |
| `config_` / `project_` | 配置/项目 | `config_load_success` |
| `ai_` | AI 相关 | `ai_analysis_start` |

### 4.2 格式化占位符

翻译值中使用 Go 标准 `fmt` 占位符（`%s`, `%v`, `%d` 等），中英文必须保持一致的占位符顺序和数量。

```go
// zh.go
"scene_create_failed": "❌「%s」场景创建失败\n %s\n",
// en.go
"scene_create_failed": "❌ Scene \"%s\" creation failed\n %s\n",
```

## 6. 与前端 i18n 的关系

```
┌─────────────────────────────────────────────────────┐
│                    用户界面 (前端)                     │
│  frontend/src/lib/i18n.js                           │
│  - 300+ 翻译键 (zh/en)                               │
│  - Svelte $derived 响应式                             │
│  - localStorage 缓存                                 │
├─────────────────────────────────────────────────────┤
│              SetLanguage(lang) / GetLanguage()       │
│              ↕ Wails binding                         │
├─────────────────────────────────────────────────────┤
│                    后端 (Go)                          │
│  i18n/i18n.go + i18n/zh.go + i18n/en.go            │
│  - ~90 翻译键 (CLI 消息 + 系统通知 + 后端日志)          │
│  - goroutine 安全                                    │
│  - gui_settings.json 持久化                          │
├─────────────────────────────────────────────────────┤
│                    CLI 模式                           │
│  REDC_LANG 环境变量 / 系统 locale 自动检测               │
└─────────────────────────────────────────────────────┘
```

**关键点：前端和后端各自维护独立的翻译表，通过 `SetLanguage` 保持语言一致。**

前端翻译键用于 UI 控件（按钮、标签、对话框文本），后端翻译键用于运行时消息（日志、通知、命令行输出）。两者不共享键名，无需同步翻译表。

## 7. 添加新翻译的步骤

1. 在 `i18n/zh.go` 的 `zhMessages` 中添加中文 key-value
2. 在 `i18n/en.go` 的 `enMessages` 中添加对应的英文 key-value
3. 在代码中使用 `i18n.T("key")` 或 `i18n.Tf("key", args...)`
4. 确保中英文的 `%s`, `%v` 等格式占位符数量和顺序一致

**示例：添加一条新消息**

```go
// i18n/zh.go
"template_not_found": "模板「%s」不存在",

// i18n/en.go
"template_not_found": "Template \"%s\" not found",

// 使用处
gologger.Error().Msgf(i18n.Tf("template_not_found", name))
```

## 8. 迁移指南：将 app.go 硬编码中文迁移到 i18n

### 8.1 背景

`app.go` 文件约 5300+ 行，包含 **约 100+ 处中文硬编码**，主要分布在：

- `runtime.LogInfof/LogErrorf` 调用
- `runtime.EventsEmit(ctx, "log", ...)` 推送的日志消息
- `fmt.Errorf(...)` 返回的错误信息
- 直接 `return "中文消息"` 的函数返回值

### 8.2 迁移步骤

#### 第一步：搜索所有中文字符串

在 `app.go` 中搜索包含中文的字符串。可以用以下正则：

```bash
grep -n '[\x{4e00}-\x{9fff}]' app.go
```

#### 第二步：为每个中文字符串创建翻译 key

按照命名规范（见第 4 节）为每个字符串命名。建议按 app.go 中的函数分组：

| 函数 / 区域 | 建议 Key 前缀 | 示例 |
|------------|-------------|------|
| `startup()` / 初始化 | `app_init_` | `app_init_config_success` |
| 场景操作 (Start/Stop/Kill) | `app_scene_` | `app_scene_start_failed` |
| SSH 操作 | `app_ssh_` | `app_ssh_connect_failed` |
| 版本检查 | `app_version_` | `app_version_check_failed` |
| AI 分析 | `app_ai_` | `app_ai_analyzing` |
| 自定义部署 | `app_deploy_` | `app_deploy_create_success` |
| 编排 | `app_compose_` | `app_compose_up_failed` |
| 成本估算 | `app_cost_` | `app_cost_calc_failed` |
| 模板管理 | `app_tmpl_` | `app_tmpl_pull_success` |
| 定时任务 | `app_schedule_` | `app_schedule_create_success` |
| 文件管理 | `app_file_` | `app_file_upload_failed` |

#### 第三步：添加到翻译表

在 `i18n/zh.go` 和 `i18n/en.go` 底部添加新的 key-value：

```go
// i18n/zh.go - 在 zhMessages map 中追加
"app_init_config_success": "配置加载成功",
"app_init_project_success": "项目加载成功",
// ...

// i18n/en.go - 对应追加
"app_init_config_success": "Configuration loaded successfully",
"app_init_project_success": "Project loaded successfully",
// ...
```

#### 第四步：替换代码中的硬编码

将原来的：
```go
runtime.LogInfof(a.ctx, "配置加载成功")
```

替换为：
```go
runtime.LogInfof(a.ctx, "%s", i18n.T("app_init_config_success"))
```

对于带参数的：
```go
// 原来
runtime.LogErrorf(a.ctx, "Profile 初始化失败: %v", err)

// 替换为
runtime.LogErrorf(a.ctx, "%s", i18n.Tf("app_init_profile_failed", err))
```

对于 EventsEmit 日志推送：
```go
// 原来
runtime.EventsEmit(a.ctx, "log", "配置加载成功")

// 替换为
runtime.EventsEmit(a.ctx, "log", i18n.T("app_init_config_success"))
```

对于 fmt.Errorf：
```go
// 原来
return fmt.Errorf("未知操作: %s", action)

// 替换为
return fmt.Errorf("%s", i18n.Tf("app_unknown_action", action))
```

#### 第五步：验证

```bash
# 编译检查
go build ./...

# vet 检查（i18n 动态字符串会产生 non-constant format string 警告，属于正常现象）
go vet ./...
```

### 8.3 注意事项

1. **不要翻译日志中的技术标识**：如 case ID、模板名称、IP 地址等保持原样作为参数传入
2. **EventsEmit 的 event name 不翻译**：`"log"`, `"refresh"` 等事件名是前端监听的，不做翻译
3. **一对一替换**：每次只处理一个中文字符串，替换后即编译验证，避免批量替换出错
4. **保持占位符一致**：中英文翻译中的 `%s`, `%v`, `%d` 数量和顺序必须一致
5. **`runtime.Log*f` 的特殊处理**：由于 `go vet` 检查，建议使用 `runtime.LogInfof(ctx, "%s", i18n.T("key"))` 的写法

### 8.4 预估工作量

| 类别 | 预估数量 | 优先级 |
|------|---------|-------|
| `runtime.EventsEmit` 日志推送 | ~40 处 | P0（用户可见） |
| `runtime.Log*f` 后端日志 | ~30 处 | P1 |
| `fmt.Errorf` 错误消息 | ~20 处 | P1 |
| 函数返回的字符串 | ~15 处 | P2 |

**建议优先处理 P0（EventsEmit），因为这些消息直接展示给用户。**

## 9. 其他模块的 i18n 迁移

除了 `app.go`，以下 `mod/` 目录中的文件也包含中文硬编码，可按需迁移：

| 文件 | 说明 |
|------|------|
| `mod/modules.go` | 模块钩子系统的日志 |
| `mod/case.go` | 场景操作的日志和错误 |
| `mod/terraform.go` | Terraform 执行的日志 |
| `mod/compose/` | 编排系统的日志 |
| `mod/template_manager.go` | 模板管理的日志 |
| `mod/custom_deployment.go` | 自定义部署的日志 |

迁移方式与 `app.go` 完全一致，使用 `red-cloud/i18n` 包的 `T()` / `Tf()` 函数即可。
