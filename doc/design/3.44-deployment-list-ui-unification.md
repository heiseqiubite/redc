# 3.44 自定义部署列表 UI 统一与 MCP 工具安全增强

## 概述

v3.1.4 对自定义部署列表进行了全面的 UI 重构，将自定义 CSS 迁移到 Tailwind 工具类，与预定义场景（Cases.svelte）保持一致的视觉风格。同时增强了 MCP 工具的安全性，防止 AI Agent 误操作。

---

## 1. 自定义部署列表 Tailwind 迁移

### 问题
CustomDeploymentList.svelte 使用约 300 行自定义 `<style>` CSS（`.deployment-list`、`.list-header`、`.deployment-table` 等），而 Cases.svelte 已全面使用 Tailwind 工具类。两个列表视觉风格不统一，维护成本高。

### 方案
将 CustomDeploymentList 的模板和样式完全对齐 Cases.svelte 的设计模式。

## 2. 搜索与状态筛选

### 问题
旧列表没有搜索功能，用户在大量部署中难以快速找到目标。

### 方案
在列表头部添加搜索框和状态标签页，与 Cases.svelte 保持一致。

### 新增状态变量

```javascript
let deploymentSearch = $state('');           // 搜索关键词
let deploymentStatusFilter = $state('all');  // 状态筛选：all/running/stopped/error

// 状态计数（derived）
let deploymentStatusCounts = $derived({
  all: filteredDeployments.length,
  running: filteredDeployments.filter(d => d.state === 'running').length,
  stopped: filteredDeployments.filter(d => d.state === 'stopped' || d.state === 'pending').length,
  error: filteredDeployments.filter(d => d.state === 'error').length,
});

// 最终可见列表（derived，结合标签筛选 + 搜索 + 状态筛选）
let visibleDeployments = $derived(
  filteredDeployments.filter(d => {
    const matchSearch = !deploymentSearch ||
      d.name.toLowerCase().includes(deploymentSearch.toLowerCase()) ||
      d.id.toLowerCase().includes(deploymentSearch.toLowerCase()) ||
      (d.template_name || '').toLowerCase().includes(deploymentSearch.toLowerCase());
    const matchStatus = deploymentStatusFilter === 'all' ||
      (deploymentStatusFilter === 'running' && d.state === 'running') ||
      (deploymentStatusFilter === 'stopped' && (d.state === 'stopped' || d.state === 'pending')) ||
      (deploymentStatusFilter === 'error' && d.state === 'error');
    return matchSearch && matchStatus;
  })
);
```

### 筛选链路

```
deployments（原始列表）
  → filteredDeployments（标签筛选，通过 selectedTag）
    → visibleDeployments（搜索 + 状态筛选）
```

### 搜索字段
- 部署名称（`d.name`）
- 部署 ID（`d.id`）
- 模板名称（`d.template_name`）

### 状态标签页
| 标签 | 匹配状态 |
|------|---------|
| 全部 | 所有 |
| 运行中 | `running` |
| 已停止 | `stopped`、`pending` |
| 错误 | `error` |

每个标签右侧显示对应计数数字。

## 3. 批量操作改进

### 问题
旧设计需要先点击「批量操作」按钮进入批量模式，再勾选复选框。Cases.svelte 的设计是复选框始终可见，选中后自动显示批量操作栏。

### 方案
- 移除 `batchMode` 状态变量和 `toggleBatchMode()`、`selectAll()`、`deselectAll()` 函数
- 复选框列始终显示（表头有全选 checkbox）
- 当 `selectedDeploymentIds.size > 0` 时自动显示蓝色批量操作栏
- 批量操作栏包含：已选计数 + 清除选择链接 + 批量启动/停止/删除按钮

### 行点击行为变更
```javascript
// 旧逻辑：batchMode 下点击行切换选择，否则展开详情
function handleSelectDeployment(deployment) {
  if (batchMode) {
    toggleDeploymentSelection(deployment.id);
  } else {
    toggleDeploymentExpand(deployment.id);
  }
}

// 新逻辑：点击行始终展开/折叠详情，选择通过复选框完成
function handleSelectDeployment(deployment) {
  toggleDeploymentExpand(deployment.id);
}
```

复选框的 `<td>` 添加 `onclick={(e) => e.stopPropagation()}` 防止点击复选框时触发行展开。

## 5. MCP 工具安全增强

### 问题
AI Agent 使用 `stop_case`/`kill_case` 工具时，可能误操作非当前对话创建的场景，或对已停止的场景执行无意义操作。实际案例：用户要求"关闭所有"，Agent 关闭了 14 个场景（仅 4 个是当前对话创建的）。

### 根因分析
1. `toolStopCase` 和 `toolKillCase` 没有状态前置检查
2. 工具描述没有说明状态限制
3. AI 系统提示词缺少作用域限制规则

### 修复 1：状态守卫

**文件**: `mod/mcp/mcp.go`

```go
// toolStopCase — 只允许停止 running 状态的场景
func toolStopCase(...) {
    c, _ := redc.LoadCaseByKeyword(id)
    if c.State != redc.StateRunning {
        return fmt.Errorf("场景 %s 当前状态为 %s，只有 running 状态的场景才能停止", id, c.State)
    }
    // ...
}

// toolKillCase — 不允许强制终止已 stopped 的场景
func toolKillCase(...) {
    c, _ := redc.LoadCaseByKeyword(id)
    if c.State == redc.StateStopped {
        return fmt.Errorf("场景 %s 已经是 stopped 状态，无需强制终止", id)
    }
    // ...
}
```

### 修复 2：工具描述增强

在 MCP tool schema 的 `description` 中添加状态前置条件：
- `stop_case`: "... 只对 running 状态的场景有效"
- `kill_case`: "... 不适用于已 stopped 的场景"

### 修复 3：系统提示词增加规则

**文件**: `mod/ai/prompt.go`

在 `工作原则` 部分新增两条规则：
```
3. 作用域限制：只操作当前对话中创建的场景，不要批量操作未确认的场景
4. 状态感知：执行 stop 前确认场景为 running，执行 kill 前确认场景不是 stopped
```

### 状态常量参考

| 常量 | 值 | 含义 |
|------|-----|------|
| `redc.StateCreated` | `"created"` | 已创建未启动 |
| `redc.StateRunning` | `"running"` | 运行中 |
| `redc.StateStopped` | `"stopped"` | 已停止 |
| `redc.StateError` | `"error"` | 错误 |

## 注意事项

1. **筛选链路顺序**：`deployments → filteredDeployments（标签）→ visibleDeployments（搜索+状态）`。表头全选 checkbox 和 colspan 使用 `visibleDeployments`，确保只操作当前可见的部署。
2. **colspan 值**：表格列数为 9（checkbox + 名称 + 模板 + 云厂商 + 地域 + 状态 + 创建时间 + 操作），标签编辑行和展开行的 colspan 需匹配。
3. **batchMode 移除**：`getBatchSelection()` 导出函数中 `mode` 改为 `selectedDeploymentIds.size > 0`，保持 API 兼容。
4. **模态框不动**：line 1400+ 的所有模态框（SSH、Schedule、Clone、PlanPreview、BatchConfirm）已使用 Tailwind，本次不修改。
5. **右键菜单不动**：底部 contextMenu 区域已使用 Tailwind，保持原样。
