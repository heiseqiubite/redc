# 3.30 Spot 实例销毁监控

## 概述

抢占式实例（Spot Instance）会在价格超出出价或资源不足时被云厂商自动回收。本功能通过后台 TCP 连通性探测，监控运行中的 Spot 实例是否被回收，并在回收时通知用户、更新场景状态。

## 架构设计

### 监控方式

**TCP SSH 端口探测（主方案）**

| 项目 | 说明 |
|------|------|
| 探测目标 | `public_ip:22`（SSH 端口） |
| 探测间隔 | 每 120 秒一轮扫描 |
| 重试机制 | 3 次连续尝试，每次 timeout 10s，间隔 15s |
| 判定逻辑 | 3 次全部失败 → 判定为已回收 |
| 去重保护 | 同一 case 只告警一次（`alerted` map） |
| 启动延迟 | 应用启动后 60s 开始首次扫描 |

选择 TCP 探测而非云 API 的原因：
- 不需要额外的云厂商 API 权限
- 跨云通用（阿里云、AWS、火山引擎等一致逻辑）
- 实现简单，无需集成多套 SDK

### 流程图

```
App.startup()
  └─ SpotMonitor.Start()
       └─ goroutine loop (每 120s)
            ├─ LoadProjectCases()
            ├─ 过滤: state == "running" && isSpot == true && !alerted
            ├─ 获取 public_ip (TfOutput)
            └─ probeSSH(ip) — TCP Dial :22 × 3
                 ├─ 成功 → 跳过
                 └─ 失败 → handleTerminated()
                            ├─ case.StatusChange("terminated")
                            ├─ EventsEmit("spot-terminated", {caseId, caseName, template})
                            ├─ emitLog("⚠️ 抢占式实例已被回收: xxx")
                            └─ emitRefresh()

Frontend (Cases.svelte)
  └─ EventsOn("spot-terminated")
       ├─ 显示 Toast 通知（10s 自动消失）
       └─ refresh() 刷新列表
```

## 关键代码

### SpotMonitor 结构

```go
type SpotMonitor struct {
    app      *App
    stopCh   chan struct{}
    wg       sync.WaitGroup
    interval time.Duration
    alerted  map[string]bool  // 防重复告警
    mu       sync.Mutex
}
```

### 探测逻辑

```go
func (m *SpotMonitor) probeSSH(ip string) bool {
    addr := net.JoinHostPort(ip, "22")
    for attempt := 0; attempt < 3; attempt++ {
        conn, err := net.DialTimeout("tcp", addr, 10*time.Second)
        if err == nil {
            conn.Close()
            return true  // 可达
        }
        // 间隔 15s 重试
    }
    return false  // 3 次全部失败
}
```

### 扫描过滤条件

仅对满足以下全部条件的 case 进行探测：
1. `state == "running"`
2. `detectSpotFromTfFiles(path) == true`（复用 3.29 的 spot 检测逻辑）
3. 尚未被标记为 alerted

### Toast 通知

- 位置：右下角固定定位
- 样式：红色警告（⚠️ 图标 + 场景名称 + IP 列表）
- 持续：15 秒自动消失，可手动关闭
- 触发：收到 `spot-terminated` 事件

### 自动恢复（Spot Auto-Recovery）

#### 原理

Terraform 是声明式的。代理池模板（如 `aliyun/proxy`）使用 `count = var.node` 创建 N 台实例。当某台被回收后：
- Terraform state 仍记录 N 台
- 执行 `terraform apply` 时，Terraform 检测到实际只剩 N-1 台
- **自动创建缺失的那 1 台，补齐到 count 数量**
- 其余 N-1 台保持不动

因此，自动恢复 = 重新执行 `TfApply()`。

#### 流程

```
handleTerminated()
  ├─ 通知用户（已回收）
  └─ 检查 SpotAutoRecoverEnabled
       └─ true → attemptRecover()
            ├─ emit "spot-recovering"
            ├─ 恢复 state → running（允许 TfApply）
            ├─ c.TfApply()
            │   ├─ 成功 → ResetAlert() + emit "spot-recovered" + 通知
            │   └─ 失败 → emit "spot-recover-failed" + 通知
            └─ emitRefresh()
```

#### 设置项

- `SpotAutoRecoverEnabled`（`gui_settings.json`，默认 `false`）
- Settings 页面中，嵌套在 Spot 监控开关下方（仅监控开启时可见）
- 左侧有缩进 + 竖线，视觉上表示从属关系

#### 前端 Toast 状态

| 事件 | Toast 样式 | 自动消失 |
|------|-----------|---------|
| `spot-recovering` | 蓝色 + 旋转图标 — "正在自动恢复..." | 不消失 |
| `spot-recovered` | 绿色 + ✅ — "已自动恢复" | 10s |
| `spot-recover-failed` | 琥珀色 + ⚠️ — "自动恢复失败" | 15s |

恢复成功后自动关闭 terminated 红色 Toast。

## 扩展方向

### 云 API 精确检测（未来）

可通过各云厂商的 DescribeInstances API 精确判断实例状态：

| 云厂商 | API | 回收判据 |
|--------|-----|---------|
| 阿里云 | `DescribeInstances` | `Status != "Running"` |
| AWS | `DescribeInstances` | `State.Name != "running"` |
| 火山引擎 | `DescribeInstances` | `Status != "RUNNING"` |

需从 terraform state 中提取 instance ID，并调用对应 SDK。结构上已预留 `checkSpotCaseViaAPI()` 扩展接口。

## 注意事项

- **误报**：网络波动可能导致误判，但 3 次重试 + 每次 10s timeout 大幅降低概率
- **状态恢复**：用户重新 `start_case` 后会变回 `running`，`ResetAlert()` 可清除告警记录
- **性能**：每轮扫描只检查 running + spot 的 case，通常数量很少，不会造成性能问题
